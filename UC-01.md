# Use Case UC-01: Register new user account

**Goal in Context**: A new user creates a CMS account by providing a unique, valid email address and a secure password so they can access CMS features beyond public announcements.  
**Scope**: Conference Management System (CMS)  
**Level**: User Goal  
**Primary Actor**: New User (Public user becoming an authorized user)  
**Secondary Actors**: Database, Email Service  
**Trigger**: The user requests to be registered on the system.

## Success End Condition

- Pending registration is converted into an active user account only after email verification succeeds.
- The activated user is redirected to login and can authenticate immediately.

## Failed End Condition

- No active account is created.
- User remains in registration/login flow with corrective guidance.

## Preconditions

- The CMS registration feature is available.
- User is unauthenticated.

## Main Success Scenario

1. User selects registration option.
2. System presents registration form.
3. User enters required fields (`email`, `password`, `confirm password`) and submits.
4. System validates required fields, email format, uniqueness, and password policy.
5. System creates pending registration and issues verification token.
6. System sends verification email.
7. User opens valid tokenized verification link within 24 hours.
8. System verifies token, activates account, and redirects to login.
9. User logs in successfully.

## Extensions

- **4a: Invalid email format**
  - **4a1**: System returns field-level email error and keeps user on registration page.
- **4b: Duplicate email (active or pending)**
  - **4b1**: System rejects request and returns duplicate-email guidance.
- **4c: Weak/non-compliant password**
  - **4c1**: System returns password-rule violations.
- **4d: Missing required fields**
  - **4d1**: System returns all missing/invalid fields in stable order.
- **5a: Concurrent same-email registration attempt**
  - **5a1**: System atomically rejects second attempt to prevent duplicate pending records.
- **6a: Verification link expired (`>=24h`, server UTC)**
  - **6a1**: System rejects verification and offers resend subject to limits.
- **6b: Verification token invalid/tampered/already-used**
  - **6b1**: System rejects verification; account remains inactive.
- **6c: Verification resend rate-limited**
  - **6c1**: System enforces max 3 resend attempts per 24h and 60s cooldown.
- **7a: Login attempted before verification**
  - **7a1**: System denies login, shows verification reminder, and offers resend when eligible.
- **7b: Pending registration age `>=7 days`**
  - **7b1**: System marks pending registration expired; login/resend denied; user must start new registration.
- **6d: Email delivery timeout/failure**
  - **6d1**: Pending registration remains intact; system provides retry guidance.
