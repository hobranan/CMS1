# Use Case UC-21: Issue Registration Ticket Confirmation

**Goal in Context**: After successful payment confirmation, the attendee receives a ticket confirmation with reference and QR code, stored as a PDF and retrievable later.  
**Scope**: Conference Management System (CMS)  
**Level**: User Goal  
**Primary Actor**: User  
**Secondary Actors**: CMS services, data store  
**Trigger**: The primary actor initiates this use case.

## Success End Condition
* Attendee can access issued ticket PDF with reference and QR code from confirmation flow and account area.

## Failed End Condition
* No false successful issuance is shown; attendee receives explicit pending/failure guidance.

## Preconditions
* Attendee is authenticated.
* Registration payment is confirmed and recorded.

## Main Success Scenario
1. CMS confirms payment is recorded.
2. CMS generates registration ticket content.
3. CMS assigns unique ticket reference.
4. CMS generates QR code payload.
5. CMS stores ticket as PDF.
6. CMS shows ticket confirmation in UI.
7. CMS attempts notification delivery.
8. CMS keeps ticket available in account area for later retrieval.

## Extensions
* **4a**: Payment still pending
  * **4a1**: CMS does not issue ticket and shows pending guidance.
* **7a**: Delivery fails after ticket issue
  * **7a1**: CMS shows delivery failure notice, but ticket remains retrievable in account.
* **4b**: Ticket generation fails
  * **4b1**: CMS returns issuance error and no ticket is issued.
* **4c**: Ticket storage fails
  * **4c1**: CMS returns storage error and warns retrieval may be unavailable.
* **4d**: Attendee leaves page before viewing
  * **4d1**: Ticket remains retrievable from account area.
